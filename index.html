The Rule of 72

Yup, it applies to software too.

The Rule of 72

If you have X% growth over one period of time, you'll double your size in 72/X time periods.

Example

If you're growing 12% per month, you'll double your users in roughly 6 months (72/12)

Doubling every six months is a big deal

And is extremely stressful

Growing quickly gives you all sorts of new things to think about

  • Covering costs
  • Continuing growth
  • User feedback (email, Twitter, Facebook, etc)
  • Lawyers and stuff
  • Hiring

And those are just the business concerns.

As for the technology side...

You will constantly struggle with

  • Adding features
  • Feature creep
  • Adding the right features (user research)
  • Adding features correctly, so you can add more features later
  • Maintaining existing features

Not to mention

  • Keeping your servers running
  • Maintaining a clean codebase
  • Coordinating a team
  • Shipping

This talk is about all of those things

...except managing your servers. You're on your own for that one.

Oh, and lawyers. Definitely not talking about that.

What is estimation?

Heisenberg’s Uncertainty Principle

It is very difficult to make a vigorous, plausible, and job-risking defense of an estimate that is derived by no quantitative method, supported by little data, and certified chiefly by the hunches of the managers.

Fred Brooks

Estimation, according to a dictionary:

  1. A tenative evaluation or rough calculation
  2. A preliminary calculation of the cost of a project
  3. A judgement based on one’s impressions; opinion

Important distinctions

  • Estimates
  • Targets
  • Commitments
  • Plans

A working definition of a “Good Estimate”

A good estimate is an estimate that provides a clear enough view of the project reality to allow the project leadership to make good decisions about how to control the project to hit its targets.

Steve McConnell

Before we jump in

Front end development has grown up. We have a core set of practices that we try to follow.

While developing, we

  • Work locally
    (run a local server, don’t FTP up changes
  • Organize our code into distinct parts
  • Use preprocessors (they make life a lot easier)
  • Livereload your files during development
    (save precious seconds that we could be developing)
  • Version control your source files
    (seriously, this is a big deal)
  • Lint our code

Before deploying, we

  • Minify all the things!
  • Concatenate all the things!
  • Set assets to far-future cache and bust the user’s cache only when they change
  • Run tests. Run tests. And run tests again
